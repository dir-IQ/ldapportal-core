Mockup scope: the IVIA Account tab inside the existing user-edit modal (UserForm.vue), preceded by an identity header that lives above the tab strip. Sibling tabs (Attributes, Groups) keep their current behaviour — only the IVIA tab and the header are new.
Not in this mockup (deliberate): the standalone user-detail page's right rail (Profile & scope, Recent activity), the fieldset split of demographic attributes, and the Audit / Approvals tabs. Those land in a separate follow-up if the team wants the full-page layout.
Header omissions: Last login and Created … by are not surfaced because neither datapoint is exposed by LdapUser today (OpenLDAP authTimestamp / AD lastLogon and LDAP createTimestamp need a backend extension). Add them in a follow-up when the backend surfaces the fields.
Not displayed in the IVIA tab (per requirement): authority and topology — the design spec carries them; the UI does not.
Styling: approximates Tailwind 4 defaults via CDN. Production uses the project's .btn-* / .input utility classes (six state layers including Compact density) per docs/frontend-conventions.md.
Data shape: at-a-glance pills bind to fields already on the getStatus response (linked / acctValid / validUntil / daysRemaining / pwdValid / pwdLastChanged). Lifecycle events come from the existing audit-log API filtered to directoryId + targetDn, with client-side filter for detail.source === 'ivia'.
All identities, DNs, emails, dates are fictional.
